# 规划.md

## 功能

Skills 桌面端、项目管理、自定义 Skill 上传与社区模块

## 功能描述

当前分支 `main-skills-project` 的目标，是把桌面端 Skills 能力从“单纯浏览列表”推进到“可搜索、可安装、可管理项目、可上传自定义 Skill、可进入社区交流”的完整桌面入口。现阶段已经完成了 Skills 首页主路径、安装弹窗、项目管理、自定义上传、社区入口与社区 MVP 骨架，但还没有把所有行为都接到真实 Mongo 写入链路。

这份规划文档的作用不是重新描述愿景，而是把已经完成的能力、当前代码基线、Mongo / Egg / Flutter 三层命名关系，以及接下来分阶段要继续补的内容沉淀下来。这样后续继续推进时，可以直接按文件、模型名、collection 和页面入口往下查，不需要每次重新梳理上下文。

## 用户故事

作为桌面端用户  
我想浏览、筛选、安装并管理各类 Skill，同时在社区里查看讨论、进入帖子详情、沉淀安装经验  
以便把 Skills 变成一个可持续使用、可协作、可扩展的桌面工作台

## 问题陈述

问题 1: Skills 首页之前虽然已经有设计方向，但缺少稳定的分页、项目管理、自定义上传和社区入口整合。

问题 2: 社区入口最初只是导航文案，占位存在，但点击没有反应，也没有列表到详情的真实页面链路。

问题 3: Mongo / Egg / Flutter 三层命名和 collection 绑定很容易走偏，如果没有统一规划，后续继续接发帖、回帖、关注等接口时很容易再次出现“库里有数据但页面查不到”的问题。

## 解决方案陈述

解决方案 1: 以桌面端 `SkillHubSection` 为中心，继续维持腾讯 SkillHub 风格的 Skills 首页，收口分类、分页、详情、安装、项目管理和自定义上传的主流程。

解决方案 2: 将社区模块先按“贴吧式论坛 MVP”落地，而不是一步做成实时聊天室。优先打通社区入口、板块列表、帖子列表、帖子详情、回复流和发帖/回帖的骨架交互。

解决方案 3: 先把社区相关的 Flutter Model、Mongo 查询 Manager、Egg Model 与 collection 显式对齐，确保后续接真实 Mongo 数据时是在现有结构上增量推进，而不是返工重写。

## 要遵循的模式

### Mongo 查询层缓存模式

来自 [SkillMongoApisManager.dart](/Users/linzhibin/Desktop/code/MangaKingFlutter/lib/com/timehello/common/database/apis/SkillMongoApisManager.dart) 与 [SkillCommunityMongoApisManager.dart](/Users/linzhibin/Desktop/code/MangaKingFlutter/lib/com/timehello/common/database/apis/SkillCommunityMongoApisManager.dart):

- Manager 内维护本地列表缓存
- 查询成功后同步到 `GlobalStateEnv`
- 页面优先消费统一 getter 和缓存，不在 UI 里直接拼底层字段
- count 必须走真实查询，不允许写死常量

### 桌面端页面结构模式

来自 [SkillHubSection.dart](/Users/linzhibin/Desktop/code/MangaKingFlutter/lib/com/timehello/page/skillhub/SkillHubSection.dart) 与 [SkillCommunityHomePage.dart](/Users/linzhibin/Desktop/code/MangaKingFlutter/lib/com/timehello/page/skillcommunity/SkillCommunityHomePage.dart):

- 顶部导航保持桌面端统一风格
- 主区域以 `Center + ConstrainedBox` 控制最大宽度
- 复杂页面拆成私有组件，关键位置加中文注释说明“为什么这样做”

## 设计决策

决策 1: 社区第一版按论坛而不是实时聊天室实现  
原因: 帖子和回复更适合沉淀安装经验、使用教程和问题排查内容，也更符合当前 Skill 生态阶段。

决策 2: 社区查询层先保留 Mongo 优先 + mock 兜底  
原因: 可以保证入口先稳定交付，同时不阻塞后续接真实数据。

决策 3: 继续强调 Mongo 三层对齐  
原因: 当前系统里 className、collection、Flutter modelName 很容易混淆，文档必须持续把这一层写明。

## 潜在问题

问题 1: Egg 虽然已有 community models，但如果服务未重启，`/api/1/classes/...` 仍可能查不到新类。  
解决方案: 每次新增 model 后，重启 Egg 并用 classes 接口做一次显式验证。

问题 2: 社区首页当前仍可能因为没有真实数据而回退到 mock。  
解决方案: 下一阶段优先补真实板块、帖子、回复集合数据。

问题 3: 社区发帖 / 回帖当前只做了本地体验。  
解决方案: 接下来优先把写入链路补齐，再把 UI 状态刷新改成依赖真实 Mongo 返回。
