## 推荐交互流程

录音交互不要拆成“录完立刻切旧播放器再保存”的两段式流程。推荐合并为一个弹窗：

1. 用户点击开始录音。
2. 本地临时目录生成录音文件。
3. 用户点击结束录音。
4. 弹窗内显示回放预览、录音信息、重新录制。
5. 用户点击 `Confirm` / `确认`。
6. 确认后才上传 OSS 并写业务数据。

这样可以避免用户还没试听就生成服务端数据，也方便重录和取消。

## 本地录音实现要点

录音文件推荐写入临时目录：

```dart
final dir = await getTemporaryDirectory();
final path = p.join(
  dir.path,
  'audio_${DateTime.now().millisecondsSinceEpoch}.m4a',
);
await audioRecorder.start(config, path: path);
```

停止录音时要先缓存当前时长，再调用 `stop()`：

```dart
final int duration = _recordDuration;
final path = await audioRecorder.stop() ?? "";
```

原因是 `record` 的状态流可能在 stop 后触发清理逻辑，如果先依赖状态回调再取时长，容易得到 0 秒。

## 回放预览实现要点

录音完成后先不要上传。把以下状态保存在当前组件内：

- 本地录音路径
- 录音时长
- 录音完成时间
- 播放进度

回放使用：

```dart
await _audioPlayer.play(ap.DeviceFileSource(path));
```

用户点击重新录制时，需要停止播放器、清空当前录音状态，再开始新的录音。

## 保存与上传

录音确认保存时统一走阿里云 OSS 直传封装：

```dart
File file = File(path);
String url = await AliyunStoreManager.getInstance().uploadFileByFilePath(
  docType: DocType.audio,
  path: file.path,
  fileName: Utility.getUUID(),
);
```

保存业务数据时至少保留：

```dart
{
  "file": file.lengthSync(),
  "duration": duration,
  "url": url,
  "localUrl": file.path
}
```

## 不同业务模式

`RecordPage2` 当前根据 `RichTextModeEnum` 进入不同保存分支：

- `RichTextModeEnum.diary`：弹标题输入框，写入语音日记 timeline。
- `RichTextModeEnum.getUrl`：上传后通过 `onSubmit` 把 OSS URL、本地路径、时长、文件大小回传给外层。
- 其他模式：默认写入语音笔记 timeline，并通过 `onSubmit` 回传。

修改录音功能时，不要只改 UI。必须同时确认这三类保存分支是否仍然符合预期。

## 国际化规则

录音页新增文案必须先写入：

- `lib/l10n/intl_*.arb`

再生成：

- `lib/generated/*`

业务代码统一使用：

```dart
getI18NKey().xxx
```

不要在录音页面里混用 `S.of(context)`。

## 常见坑

- 不要录音结束后立即上传，应该等用户确认。
- 不要继续复用旧的灰色全屏 `AudioPlayerWidget` 作为录音保存前的唯一预览。
- 不要忘记取消 `Timer`、录音状态流、振幅流、播放器进度流。
- 不要把 `DocType.audio` 漏掉，否则音频可能上传到错误目录。
- 不要只保存 OSS URL，建议同时保存 `duration`、`file`、`localUrl`。
- 不要在页面里重写 OSS 上传逻辑，上传规则看 [阿里云的方式.md](/Users/linzhibin/Desktop/work/project/flutter/efficientTimeFinal/efficientTime5/efficientTime/docs/知识库/阿里云的方式.md)。
